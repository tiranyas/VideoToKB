1. Rewrite `src/app/page.tsx` as a client component ("use client") that orchestrates the full user flow:

   State management:
   - `videoUrl: string` -- bound to UrlForm
   - `template: string` -- bound to UrlForm dropdown
   - `isProcessing: boolean` -- disables form during pipeline execution
   - `steps: Array<{ step: PipelineStep, status: StepStatus, message?: string }>` -- initialized with all 4 steps as 'pending'
   - `error: string | null` -- error message from pipeline
   - `article: string` -- generated article content
   - `showResult: boolean` -- toggles between form/progress view and article view

2. Implement `handleSubmit(videoUrl, template)`:
   - Set isProcessing true, clear error, reset steps to pending, clear article
   - Use `fetch('/api/process', { method: 'POST', body: JSON.stringify({ videoUrl, template }) })`
   - Read response body as ReadableStream (NOT EventSource -- research says POST requires fetch)
   - Parse SSE events from stream (buffer-based parsing, split on '\n\n', extract 'data: ' prefix)
   - For each ProgressEvent:
     - Update the corresponding step in the steps array
     - If step is 'error', set error state with the message
     - If step is 'done' and has article, set article state and showResult true
   - Handle fetch errors / stream interruption: set error to "Connection lost. Please try again."
   - Set isProcessing false when stream ends
   - Follow the SSE consumption pattern from research exactly

   - App title "VideoToKB" as h1 at top, with subtitle "Turn video recordings into KB articles"
   - If !showResult: render UrlForm + ProgressDisplay (show progress only when isProcessing or steps have non-pending status)
   - If showResult: render ArticleView + a "Start Over" button that resets all state
   - Clean, minimal design -- no sidebar, no nav, no header bar (UILP-02)
